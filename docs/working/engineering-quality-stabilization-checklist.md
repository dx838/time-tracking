# 工程质量稳定期执行清单

## 1. 文档定位

本文是 `Time Tracker` 当前阶段的工程质量全流程执行文档。

它服务于：

- [`../engineering-quality-target.md`](../engineering-quality-target.md)
- [`../code-quality-target.md`](../code-quality-target.md)
- [`../performance-target.md`](../performance-target.md)
- [`../reliability-and-validation-target.md`](../reliability-and-validation-target.md)
- [`../architecture.md`](../architecture.md)

它不是长期母文档，也不是理念说明，而是一份可以逐项勾选、逐阶段推进、最终归档的执行清单。

本文的目标是：

- 把“工程质量提升”拆成可落地的阶段任务
- 让后续协作者能清楚知道当前推进到哪一步
- 避免每次都重新判断优先级、顺序和验证门槛
- 为后续沉淀长期 `engineering-quality.md` 母文档积累真实规则

当本文与长期规则文档冲突时，以 [`../architecture.md`](../architecture.md) 和 [`../engineering-quality-target.md`](../engineering-quality-target.md) 为准；本文应随之更新。

## 2. 使用方式

### 2.1 勾选规则

一个任务只有在下面条件都满足时才可以勾选：

- 代码已经落地
- 相关导入、调用路径或目录落点已经收口
- 没有引入新的边界回流
- 对应的最低验证已经完成
- 如有必要，相关文档也已经同步

### 2.2 执行粒度

本文故意把任务拆细。

默认原则是：

- 一次只推进一个小阶段或一个热点
- 先建立基线，再做高风险优化
- 先补验证保护，再做深整理
- 先解决真实热点，再讨论普遍整洁
- 先保证可靠，再追求更快

### 2.3 允许调整

本文写的是推荐顺序和推荐落点，不是机械脚本。

如果执行过程中发现：

- 更小的安全切分路径
- 更合理的 owner
- 更适合先做的低风险前置项

可以调整实现顺序，但不要改变这轮执行的总体意图：

- 先把可靠性和验证立住
- 再收代码质量热点
- 最后在有测量依据的前提下推进性能改进

### 2.4 完成判断

“完成当前阶段的工程质量执行”不等于仓库已经完美，而是至少满足：

- 已经建立统一质量 gate
- 当前阶段首批可靠性热点有更硬的验证保护
- 当前阶段首批代码质量热点完成有效收口
- 性能讨论开始建立在统一口径上
- 团队已经积累出可沉淀为长期规则的真实经验

## 4. 执行顺序总览

本文默认按下面顺序推进：

1. 建立共同基线与统一 gate
2. 先补可靠性与验证
3. 再收代码质量热点
4. 再做有测量依据的性能基线与低风险优化
5. 最后做全局收口、归档与长期规则判断

这不是硬性 waterfall。
如果某次改动只触及其中一个小块，可以局部推进，但不要颠倒总体优先级。

## 6. Phase 1：可靠性与验证先立住

本阶段目标：

- 先让“可被长期信任”成为工程事实
- 先补风险保护，再做深层重构和性能动作

### 6.1 tracking 主链可信性

- [ ] 列出 tracking 主链的关键行为不变量
- [ ] 对照现有测试，标出已经有保护的环节
- [ ] 标出仍缺直接验证的主链风险点
- [ ] 补齐 lifecycle / replay / transition / cleanup 的关键验证缺口
- [ ] 明确今后任何 tracking 主链高风险改动都必须跑的固定验证组合

### 6.2 读模型与 replay 可信性

- [ ] 审视 dashboard / history 读模型是否已有足够稳定的 replay 保护
- [ ] 标出 replay 能覆盖的范围和不能覆盖的范围
- [ ] 对高风险读模型变更补上更接近真实行为的验证
- [ ] 明确 session 编译、formatting、feature 私有拼装分别由谁负责验证

### 6.3 数据边界与恢复安全

- [ ] 梳理 backup / restore / cleanup 的风险点
- [ ] 梳理数据边界相关改动默认应追加的验证动作
- [ ] 确认数据库边界变更不会只靠人工验证放行
- [ ] 为恢复路径和清理路径建立更直接的保护意识

### 6.4 前端热点行为保护

- [ ] 在深度整理 `AppMapping` 前，先确认关键行为场景已被覆盖
- [ ] 在深度整理 `Settings` 前，先确认关键行为场景已被覆盖
- [ ] 标记哪些前端热点行为目前仍主要依赖人工回归
- [ ] 对“高频会改、容易误伤”的交互补最小但直接的保护

### 6.5 发布前硬门槛

- [ ] 明确当前阶段的默认发布前最低门槛
- [ ] 确认 `npm test`、`npm run test:replay`、`npm run build` 是当前默认底线
- [ ] 确认哪些发布场景必须追加 `cargo check`
- [ ] 确认哪些发布场景必须追加 changelog / release 相关验证
- [ ] 文档化“不满足最低门槛时不能视为可发布”

### 6.6 阶段验收

- [ ] tracking 主链高风险改动已经有固定验证组合
- [ ] replay / 生命周期 / 恢复路径不再明显裸奔
- [ ] 前端热点重构前已有最低行为保护
- [ ] 发布前验证门槛已经被明确写死

## 7. Phase 2：代码质量热点收口

本阶段目标：

- 降低热点文件的认知负担
- 让复杂度更稳定地留在真实 owner 内

### 7.1 前端热点页面收口

- [ ] 先梳理 `AppMapping.tsx` 当前承担的职责
- [ ] 明确哪些职责属于页面壳层，哪些属于 feature 内部逻辑
- [ ] 把可安全拆出的局部 UI、hooks、服务留在 classification owner 内
- [ ] 确保拆分不是把复杂度转移到新的模糊共享层
- [ ] 确认 `AppMapping` 改动后行为契约没有被悄悄改掉

- [ ] 先梳理 `Settings.tsx` 当前承担的职责
- [ ] 明确哪些职责属于设置展示，哪些属于设置编排或持久化调用
- [ ] 优先拆出 owner 清楚、状态边界清楚的局部块
- [ ] 避免把设置 feature 的私有逻辑重新塞回 `shared` 或 `app`
- [ ] 确认 `Settings` 改动后交互契约保持稳定

### 7.2 壳层与编排热点收口

- [ ] 审视 `src/app/services/*` 当前是否在继续变成吸力层
- [ ] 标记哪些服务属于真正的 app 编排，哪些已经更像 feature 或 platform owner
- [ ] 继续收薄只因“方便”而停留在 app 层的逻辑
- [ ] 保证 `app/*` 继续回到壳层与跨 feature 编排身份

### 7.3 共享逻辑热点收口

- [ ] 审视 `src/shared/lib/sessionReadCompiler.ts` 的真实职责边界
- [ ] 区分共享 session 编译核心与 feature 私有读模型拼装
- [ ] 确认没有把 feature 私有 formatting 或页面语义重新回流到 shared
- [ ] 保持 shared 只承担稳定共享能力，而不是新的过渡桶

### 7.4 Rust tracking 热点收口

- [ ] 审视 `src-tauri/src/engine/tracking/runtime.rs` 当前剩余复杂度
- [ ] 继续把可拆职责收回正确 owner 模块
- [ ] 避免把 engine 逻辑重新推回 `app`、`commands` 或 `lib.rs`
- [ ] 保证 tracking runtime 变薄时行为语义不漂移

### 7.5 测试热点本身也要变得更可维护

- [ ] 审视 `tests/trackingLifecycle.test.ts` 是否已经过厚
- [ ] 判断是否需要抽出更清楚的 fixtures、helpers 或场景分组
- [ ] 保证测试重组不会削弱主链保护力度
- [ ] 让测试文件本身也更易读、更容易定位失败原因

### 7.6 阶段验收

- [ ] 当前阶段首批热点文件职责已经更单一
- [ ] 新拆分逻辑仍留在真实 owner 内
- [ ] 高风险层没有因为“整理代码”而继续长厚
- [ ] 测试文件本身的可维护性也有所提升

## 10. 默认验证门槛

除非某项任务明确说明更低或更高门槛，否则当前默认至少执行：

- [ ] `npm test`
- [ ] `npm run test:replay`
- [ ] `npm run build`

涉及 Rust tracking 主链、数据边界或恢复路径时，额外至少执行：

- [ ] `cargo check`

如果某次改动明显触及性能热点或可靠性热点，还应追加更贴近风险的专项验证，而不是只跑基础脚本后直接结束。

## 11. 什么情况需要暂停并升级

如果某次“工程质量优化”出现下面任一情况，应暂停当前批次，升级为单独执行单或专项设计，而不是边改边扩：

- [ ] 需要跨 owner 迁移职责
- [ ] 需要引入新的共享层或平台层
- [ ] 需要大规模改变 feature 公开契约
- [ ] 需要改变产品行为，而不只是优化实现
- [ ] 需要一次性同时触及多个高风险区
- [ ] 无法在当前默认验证门槛下证明改动是安全的
- [ ] 性能优化开始依赖难以解释的特例和隐式状态
